## この章のゴール

- GitHub 上で自分にアサインされた Issue を確認できる
- Issue の内容(やるべきこと)を理解できる
- ターミナルからも Issue を確認できる

## Issue って何?

Issue は GitHub の「**やることリスト/タスクチケット**」です。バグ報告にも、機能追加の提案にも、今回のような研修タスクにも使われます。

> :::message
> 実務では、ほとんどの開発タスクが Issue から始まります。「Issue を見る → ブランチを切る → 作業する → PR を出す → Issue を閉じる」がチーム開発の基本サイクルです。今日はそのサイクルを 1 周体験してもらいます。
> :::

今日のハンズオンでは、**運営陣が事前に皆さん一人ひとりに Issue を立てています**。自分用の Issue を確認することから始めましょう。

## 方法 1: GitHub の Web で確認する

ブラウザで以下の URL にアクセスします。

```
https://github.com/Webu-Kobedenshi/git-hands-on-introductions/issues
```

または、リポジトリのページ上部にある **Issues** タブをクリックしてください。

Issue 一覧が表示されます。次に **自分にアサインされた Issue** だけに絞り込みます。

1. 検索ボックス(`is:issue is:open` と書いてある)に追記:
   ```
   is:issue is:open assignee:@me
   ```
2. Enter

これで自分にアサインされた Issue だけが表示されます。`[自己紹介] @<あなたの github-id> さん` のようなタイトルの Issue があれば成功です。

クリックして中身を見ましょう。やるべきことのチェックリストが書かれているはずです。

## 方法 2: ターミナルから確認する (`gh`)

GitHub CLI を使えば、ブラウザを開かずにターミナル上で Issue を確認できます。

```bash
gh issue list --assignee @me
```

`@me` は「ログイン中の自分自身」を意味します。

```bash
gh issue view <Issue 番号>
```

例:

```bash
gh issue view 5
```

Issue の本文がターミナルに表示されます。

> :::message
> 実務では、エディタやターミナルから離れずに Issue を読めるのは生産性的に大きいです。今日は概念を掴むため、Web 版で見ることをおすすめします。
> :::

## Issue を読み解く

自分の Issue には、以下のような情報が書かれています。

- **担当**: あなたの GitHub ID
- **チーム**: `team1` / `team2` / `team3` のどれか(これは大事!)
- **ゴールのチェックリスト**:
  - 自分専用のブランチを切る
  - `templates/self-introduction.md` をコピーして配置
  - commit / push / PR
  - 同じチームの PR にコメント
  - セルフマージ

ここで **自分のチーム番号を確認** しておいてください。次の章以降のコマンドで使います。

> :::message
> もし「自分の Issue が見当たらない」と思ったら、運営陣 (TA) に声をかけてください。GitHub アカウント名の連絡漏れの可能性があります。
> :::

## まとめ

- Issue は「やることリスト」
- Web の Issues タブ、または `gh issue list --assignee @me` で自分のタスクを確認
- 自分のチーム番号(team1/2/3)を覚えておく

次の章では、いよいよ自分専用のブランチを切ります。
