## このハンズオンのゴール

90 分後、あなたは次のことができるようになります。

- 自分の PC にある作業を、**ブランチ** という単位で切り分けて進められる
- ファイルの変更を **コミット** として記録できる
- 自分の変更を GitHub に **push** し、**Pull Request (PR)** として出せる
- 他の人の PR に **レビューコメント** を残せる
- 自分の PR を **マージ** して、`main` ブランチに反映できる

これらは Web エンジニアやアプリ開発者の現場で **毎日** 使われる操作です。今日のハンズオンが終われば、就職した次の日から「ブランチ切って PR 出しておいて」と言われても困らなくなります。

## 何を題材にするか

題材は「**自己紹介の Markdown ファイル**」です。

このリポジトリには `self-introductions/team1`・`team2`・`team3` というディレクトリがあり、皆さんはそれぞれ自分のチームのディレクトリに `<あなたの GitHub ID>.md` というファイルを作成します。テンプレートが用意されているので、そこに名前・趣味・興味のある技術などを書いていきます。

ハンズオンが終わると、リポジトリには 14 人ぶんの自己紹介が並んでいる、という成果物ができあがります。

## 前提環境

ハンズオン開始までに以下を準備してください。

### すべての OS 共通

- **GitHub アカウント**(運営に GitHub ID を共有済み)
- **Git** インストール済み
- **GitHub CLI (`gh`)** インストール済み
- 任意のテキストエディタ(VS Code 推奨)

### Windows の方

ターミナルとしては **Git Bash** を使ってください。Git for Windows をインストールすると一緒に入ってきます。

> :::message
> :warning: コマンドプロンプトや PowerShell は使いません。本ハンズオンの手順は Git Bash 前提です。
> :::

なぜ Git Bash かというと、`cp` や `mkdir -p` といった Unix 系のコマンドが macOS / Linux と同じように動くからです。コマンドプロンプトや PowerShell では同じコマンドが動かないか、挙動が違います。Git Bash を使えば全員が同じコマンドで進められます。

### macOS / Linux の方

OS 標準のターミナルで OK です。

## 動作確認

ターミナルを開いて以下を実行し、エラーが出ないことを確認してください。

```bash
git --version
```

```bash
gh --version
```

両方ともバージョン番号が表示されればインストール OK です。「コマンドが見つかりません」と出る場合は次の章までに運営に声をかけてください。

## 詰まったときの対処

ハンズオン中に詰まったら、次の順序で対処してください。

1. **エラーメッセージをよく読む** — 英語でも、知っている単語を拾うだけで原因が分かることが多いです
2. **11 章「よく出るエラーと対処」を見る** — 典型的なエラーはここに書いてあります
3. **同じチームのメンバーに聞く** — 同じ場所で詰まっているかもしれません
4. **TA(運営陣)に声をかける** — 各チームに 1 名います

「自分の作業だけが遅れているかも」と心配しなくて大丈夫です。**詰まること自体がハンズオンの一部** です。

## なぜ Git/GitHub を学ぶのか

最後に、今日 90 分を投資する理由を一つだけ。

ソフトウェア開発では、**過去のあらゆる時点に戻れる** ことと、**複数人の変更を安全に統合できる** ことが死活的に重要です。Git はこの 2 つを実現する仕組みで、GitHub はそれをチームで使えるようにしたサービスです。

> :::message
> もし Git なしで開発したら? 「昨日のバージョンに戻したい」「A さんと B さんの変更を合わせたい」という当たり前の要求が、フォルダのコピーや手作業マージで地獄のように難しくなります。Git は、その地獄を回避するために生まれた道具です。
> :::

それでは次の章で、Git/GitHub の中身を簡単に見ていきましょう。
