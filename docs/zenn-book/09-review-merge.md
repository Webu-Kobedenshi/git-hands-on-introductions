## この章のゴール

- 同じチームのメンバーの PR を Web で開ける
- 行コメント・PR 全体コメントを残せる
- 自分の PR をセルフマージできる

## レビューのルール(今日の研修)

今日は次のルールでレビューします。

- **必須**: 自分と同じチーム(`team1` / `team2` / `team3`)のメンバーの PR に **最低 1 件** コメント
- **任意**: 他のチームの PR を見てコメントしても OK
- **マージ**: 自分の PR は **自分でマージ**(セルフマージ)

> :::message
> 実務では「他人がレビューして OK を出すまでマージできない」ルールがほとんどですが、今日は時間と人数の都合でセルフマージにします。レビューコメントの体験をすることが目的です。
> :::

## ステップ 1: チームメンバーの PR 一覧を見る

まず、いま開いている PR の一覧を確認します。

```bash
gh pr list
```

または Web で:

```
https://github.com/Webu-Kobedenshi/git-hands-on-introductions/pulls
```

自分と同じチームのメンバーが出している PR を見つけましょう。タイトルか変更ファイルのパス(`self-introductions/team2/...` など)で見分けられます。

## ステップ 2: PR を開いて中身を読む

レビューしたい PR をクリックします。3 つのタブが上部に並んでいます。

| タブ | 何が見える |
|---|---|
| **Conversation** | PR の説明・これまでのコメント |
| **Commits** | この PR に含まれる commit 一覧 |
| **Files changed** | 変更されたファイルの差分 |

**Files changed** タブを開きましょう。追加された自己紹介ファイルの差分が緑色で表示されます。

### Markdown プレビューで読む

差分表示だと「`# 自己紹介` の行が `+` 付きで表示」のように生のテキストとして見えます。せっかくの自己紹介なので **整形した状態** で読みたいですよね。

ファイル名の右にある **`...`(3 点メニュー)** をクリックして、**View file** を選んでください。Markdown が整形されたプレビュー画面に飛びます。

> :::message
> Markdown ファイルは GitHub 上で見ると、見出しが大きく表示されたり、リストが箇条書きになったりして、ぐっと読みやすくなります。レビューでは Files changed の差分とプレビューを行ったり来たりするのが普通です。
> :::

## ステップ 3: 行コメントを残す(おすすめ)

差分画面で気になる行があれば、その行に **直接コメント** を残せます。

1. **Files changed** タブに戻る
2. コメントしたい行の **左側の行番号にカーソルを合わせる** と青い `+` ボタンが出る
3. `+` ボタンをクリック
4. コメント欄が開くので、コメントを書く
5. **Add single comment** ボタンを押す

これで「自己紹介の○行目に対するコメント」として表示されます。

### コメント例

- 「同じ趣味で嬉しいです!」
- 「Unity 触ってるんですね、何作ってるんですか?」
- 「最近見たアニメ気になります」

> :::message
> 実務のレビューでは「ここはこう書いた方が読みやすいです」「この変数は別名のほうが意図が伝わります」のような **改善提案** が中心ですが、今日は気軽な反応で OK です。
> :::

## ステップ 4: PR 全体コメントを残す(任意)

行ではなく PR 全体に対するコメントは、**Conversation** タブの下部にあるコメント欄から残せます。

ここには「全体の感想」「読んだ印象」を書くと良いでしょう。

## ステップ 5: 自分の PR に戻る

レビュータイムが終わったら、自分の PR をマージします。

```bash
gh pr view --web
```

または、自分が出した PR を Web で開きます。

## ステップ 6: マージする

PR ページの下部に緑色の **Merge pull request** ボタンがあります。

1. **Merge pull request** をクリック
2. **Confirm merge** をクリック
3. 「Pull request successfully merged and closed」と表示されれば完了

> :::message
> マージ方法は 3 種類ありますが、今日は **デフォルトの「Create a merge commit」** で OK です。違いについては 12 章で軽く触れます。
> :::

### ブランチ削除(任意)

マージ後、PR ページに **Delete branch** ボタンが出てきます。役目を終えたブランチは削除して構いません。リモートのブランチが消えます。

```bash
# ローカルのブランチも消す場合
git switch main
git branch -D add-self-introduction-octocat
```

> :::message
> 削除しなくてもリポジトリは正常に動きますが、何十個も古いブランチが残っていると見通しが悪くなります。実務でもマージ後はブランチを消すのが一般的です。
> :::

## ステップ 7: main を最新にする

セルフマージしたあと、自分のローカルの `main` ブランチも最新にしておきましょう。

```bash
git switch main
```

```bash
git pull
```

これで、他の人がマージした自己紹介も含めた最新の `main` が手元に来ます。

```bash
ls self-introductions/team1
ls self-introductions/team2
ls self-introductions/team3
```

各チームのディレクトリに、メンバーぶんの `.md` ファイルが並んでいるはずです。

## ありがちなつまずき

### Q. Merge ボタンが押せない

ブランチ保護がかかっているか、コンフリクトしているかです。今日はブランチ保護をかけていないので、おそらくコンフリクトです。10 章「コンフリクト」を参照してください(ただし今日のハンズオンの設計上、コンフリクトはほぼ発生しないはずです)。

### Q. レビューコメントを間違えて送った

コメントの右上の **`...`** から **Edit** または **Delete** で修正・削除できます。

### Q. `git pull` でエラーが出た

```bash
git status
```

を実行して、コミットしていない変更が残っていないか確認してください。残っているなら commit するか `git stash` (12 章参照)してから `git pull` してください。

## まとめ

- 同じチームの PR を **Files changed** + **View file** で読む
- 行コメントは行番号の `+` ボタンから
- PR 全体コメントは Conversation タブから
- 自分の PR は **Merge pull request** ボタンでセルフマージ
- マージ後は `git pull` でローカルの `main` を最新化
