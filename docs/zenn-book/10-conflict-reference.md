## コンフリクトの正体

コンフリクト(衝突)は、**同じファイルの同じ場所を、2 人(以上)が別々に編集してマージしたとき**に発生します。

```
  Alice: 5 行目を「こんにちは」に変更
  Bob:   5 行目を「やあ」に変更
              ↓
  両方を main にマージすると…
  Git: 「5 行目どっちにすればいいの?」← これがコンフリクト
```

Git は機械なので、どちらの変更が「正しい」のか判断できません。なので、人間に判断を委ねます。これがコンフリクトです。

> :::message
> なぜ Git は自動で解決しないのか? 例えば「Alice が値を 100 → 200 に変更」「Bob が同じ値を 100 → 300 に変更」と来たら、200 と 300 のどちらが意図された値か機械には分かりません。文脈を理解する必要があるので、人間の判断が必要なのです。
> :::

## コンフリクトが **起きない** 条件

逆に、以下の場合はコンフリクトしません。

- 違うファイルを編集 → ❌ コンフリクトしない
- 同じファイルでも、違う行を編集 → ❌ コンフリクトしない
- 同じ行を、**両方が同じ内容に変更** → ❌ コンフリクトしない

> :::message
> 今日のハンズオンが「`<github-id>.md` というファイル名で各自が新規作成」設計になっているのは、**物理的にコンフリクトを発生させない** ためです。実務でも、コンフリクトを避けるためにファイル分割やコンポーネント化が行われることがあります。
> :::

## コンフリクトはいつ気付くか

コンフリクトは、次のタイミングで気付きます。

1. `git pull` したとき
2. `git merge` したとき
3. PR の Web 画面で「This branch has conflicts that must be resolved」と表示されたとき

例えばあなたの PR を見たとき、こんな表示が出ます。

```
This branch has conflicts that must be resolved
[Resolve conflicts]
```

## コンフリクトしたファイルの中身

Git はコンフリクトしたファイルに **マーカー** を埋め込んで知らせてくれます。

```
ここまでは普通の行
<<<<<<< HEAD
あなたが書いた変更
=======
相手が書いた変更
>>>>>>> add-self-introduction-bob
ここからも普通の行
```

| マーカー | 意味 |
|---|---|
| `<<<<<<< HEAD` | ここから下が **自分の変更** |
| `=======` | 区切り |
| `>>>>>>> ブランチ名` | ここまでが **相手の変更** |

## 解決のしかた(おおまかな流れ)

1. コンフリクトしたファイルをエディタで開く
2. マーカー(`<<<<<<<` `=======` `>>>>>>>`)を見つける
3. 「最終的にどう書きたいか」を考えて、マーカーごと書き換える
4. `git add <ファイル>` でステージ
5. `git commit`(マージコミットになる)

### 解決例

例えば自己紹介の趣味の行で 2 人がぶつかったとします。

修正前(コンフリクト):

```markdown
## 趣味
<<<<<<< HEAD
- 読書
=======
- ゲーム
>>>>>>> add-self-introduction-bob
```

修正後(自分の判断で統合):

```markdown
## 趣味
- 読書
- ゲーム
```

マーカーを全部削除して、最終的な形に整えます。

## VS Code を使うと楽

VS Code でコンフリクトしたファイルを開くと、上に「Accept Current Change / Accept Incoming Change / Accept Both Changes」というボタンが表示されます。

| ボタン | 動作 |
|---|---|
| Accept Current Change | 自分の変更だけ残す |
| Accept Incoming Change | 相手の変更だけ残す |
| Accept Both Changes | 両方残す(順番に並べる) |
| Compare Changes | 差分を比較表示 |

ワンクリックで解決できるので、慣れないうちは VS Code 経由がおすすめです。

## GitHub の Web 上で解決する方法

PR ページの **Resolve conflicts** ボタンを押すと、ブラウザ上のエディタが開きます。マーカーを編集して **Mark as resolved** → **Commit merge** で完了します。

> :::message
> 簡単なコンフリクトなら Web で解決するのが楽です。複雑なコンフリクトはローカルで腰を据えて解決したほうが安全です。
> :::

## まとめ

- マーカー (`<<<<<<<` 〜 `>>>>>>>`) を頼りに、最終形を自分で決める
- VS Code や Web エディタを使うと操作が楽
