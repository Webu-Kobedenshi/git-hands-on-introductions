## Git と GitHub は別物

まず一番混乱しやすいポイントから整理します。

| 名前 | 何か | どこにあるか |
|---|---|---|
| **Git** | バージョン管理の **道具(ソフトウェア)** | あなたの PC にインストールされている |
| **GitHub** | Git の中身を **預けて共有できるサービス** | インターネット上(github.com) |

Git だけあれば、自分の PC の中だけでバージョン管理はできます。GitHub は「Git のデータをクラウドに預けて、他の人と共有できるようにする場所」だと思ってください。

> :::message
> 似たようなサービスに **GitLab** や **Bitbucket** などもあります。Git は道具、GitHub はそれを使う場所、と覚えておくと混乱しません。
> :::

## リポジトリとは

**リポジトリ (repository)** は「一つのプロジェクトの保管庫」です。今回のハンズオンで皆さんが clone するのも、このリポジトリです。

リポジトリの中には:

- 実際のファイル(README.md など)
- それらの **過去のすべての変更履歴**
- ブランチや設定の情報

がまとめて入っています。フォルダのように見えて、実は変更の歴史を全部抱えているのがリポジトリです。

## Git の 4 つのエリア

Git で一番大事な概念がこれです。手を動かす章で何度も出てくるので、ここで一度イメージを作っておきましょう。

```
┌─────────────────────┐  ┌─────────────────────┐
│ ① ワーキングツリー   │  │  ④ リモート         │
│   (今編集してる    │  │  リポジトリ         │
│    ファイル)       │  │  (GitHub)         │
└─────────┬───────────┘  └──────────▲──────────┘
          │ git add                  │ git push
          ▼                          │
┌─────────────────────┐              │
│ ② ステージング       │              │
│   (commit 候補)     │              │
└─────────┬───────────┘              │
          │ git commit               │
          ▼                          │
┌─────────────────────┐              │
│ ③ ローカル          │──────────────┘
│   リポジトリ         │
│   (履歴)           │
└─────────────────────┘
```

| エリア | 中身 | どこにあるか |
|---|---|---|
| ① ワーキングツリー | エディタで編集中のファイル | あなたの PC のフォルダ |
| ② ステージングエリア | 「次の commit に入れたい変更」 | あなたの PC(隠しエリア) |
| ③ ローカルリポジトリ | 確定した履歴(commit の集まり) | あなたの PC(`.git` フォルダ) |
| ④ リモートリポジトリ | チームで共有する履歴 | GitHub(インターネット) |

エリア間を移動させるコマンドが、それぞれ:

| コマンド | やること |
|---|---|
| `git add` | ① → ② に上げる |
| `git commit` | ② → ③ に確定する |
| `git push` | ③ → ④ にアップロードする |
| `git pull` | ④ → ③ にダウンロードして取り込む |

> :::message
> なぜステージング(②)が独立しているか? 「変更を全部 commit するのではなく、関連する変更だけを選んで commit したい」場面が頻繁にあるからです。例えば「3つのファイルを直したけど、2つは機能 A の修正、1つは関係ない typo」というとき、別々の commit に分けたほうが履歴が読みやすくなります。
> :::

## コミット (commit) とは

コミットは「**ある時点でのファイル全体のスナップショット**」です。一つひとつのコミットには:

- 一意な ID(`a1b2c3d...` のようなハッシュ)
- 作者と日時
- コミットメッセージ(変更の説明)
- 直前のコミットへのリンク

がついています。コミットを並べたものが、リポジトリの **履歴** です。

```
●─●─●─●─● ← コミットがチェーン状につながっている
                              ↑ 最新
```

コミットがあるおかげで、「3 日前のあの状態に戻したい」「どこでバグが入ったか調べたい」が簡単にできるようになります。

## ブランチ (branch) とは

ブランチは「**コミットの並びにつけた名前(ラベル)**」です。前の章で出てきた図をもう一度。

```
main ─────────────────────────●─── (他の人の変更が入っていく)
       │
       └── add-self-introduction-octocat ─●─●─ (あなたの作業)
```

ブランチがあるおかげで、複数の作業が並行して進められます。`main` ブランチの履歴とは独立に、あなたのブランチで自由に commit できる、というわけです。

## GitHub の役割

GitHub は単なる「リモートリポジトリの保管場所」だけではありません。重要な機能を 3 つ覚えておきましょう。

| 機能 | 何ができるか |
|---|---|
| **Issue** | やるべきことのチケット管理 |
| **Pull Request (PR)** | ブランチの変更を main に取り込む依頼+レビューの場 |
| **コードのホスティング** | 履歴を含めてリポジトリをクラウドに保管 |

今日のハンズオンでは、**Issue を確認 → ブランチで作業 → PR を出す → レビューしてマージ** という GitHub の基本フローをひととおり体験します。

## まとめ

- Git は道具、GitHub はそれを共有する場所
- リポジトリは「ファイル + 全履歴」が入った保管庫
- Git には 4 つのエリアがあり、`add` / `commit` / `push` で順に上げていく
- コミットはスナップショット、ブランチはコミットの並びにつけた名前

ここまでが Git/GitHub の概念パートです。次の章では、いよいよ手を動かして環境を準備していきます。
